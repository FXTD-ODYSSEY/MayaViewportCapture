开发总结

  这个插件是为了完成一项测试题开发的。
  测试题详情可以参照 question 目录下 readme.md 文件

  其实之前在我也曾经开发过类似的工具
  当时也需要实现对物体不同角度输出viewport 图片以及 Arnold 渲染的图片
  不过当时没有使用 API 进行图片拼接，而是使用 ffmpeg 将多张图片合成视频
  后续的流程是通过RV播放器单帧浏览视频的流程来使用的。
  因此当时输出图片也没有用到 API 而是使用 cmds.playblast 工具

  使用 API 来处理图片是我完全没想到的方案。
  因为之前公司处理图片都是基于 Python PIL 库来实现
  但是这个库有很多 C 语言编译的 Pyd 文件，官方编译的 pyd 和 Maya 编译VS的版本不兼容
  运行起来会导致 Maya 报错。

  当时我们的解决方案是通过 subprocess 开一个外部 Python 进程对图片进行处理。
  不过开进程的开销还是挺大的，好在我们那个时候的需求也不需要等待处理的结果。
  否则Maya要未响应10多秒才能动。

  后来经过我多方的查询，在网上找到了别人编译好的 PIL 库，完美兼容 Maya
  虽然这个 PIL 库版本有点老，但是图片的基础编辑还是没有问题的
  最重要的是不用开进程，效率高。

  在收到测试题前不久，我查询文档的时候也偶然发现了 C++ 的 MImage 类
  那个时候我就意识到或许 Maya 自身就能完成一些基础的图片编辑工作。
  不过那个时候因为工作原因没有去深究 MImage 的使用。

MImage 图像处理

  直到开发这个插件才开始在网上搜索 MImage 的使用方法。
  这里面的坑还是有不少的。

  MImage 的 API 还比较简单，很多功能都需要手动实现。
  其中最为让我困惑的就是 pixels 转颜色的输出
  需要将长整形的数据调用 ctypes 获取指针数据然后转换为 字符串
  转换的字符串就是 ASCII 码 0 - 255 的范围上了，最后需要用 ord() 函数将数据转换为数值
  当然也可以在一步将数据转为 QImage 再作处理， Qt 的图片处理能力比 Maya API 强

  获取到像素颜色数据之后，后续的操作都好说。
  只是因为数据是 1 维数组，获取像素位置需要做转换，比较麻烦而已。

QImage 图像处理

  借助 QPainter 来绘制图像叠加到原图上 ，并且 MImage 的处理要简单很多。
  因为不需要单独对每一个像素进行处理了， QPainter 的 drawImage 封装处理了。

  不过 QImage 还是有些问题，没有 MImage 来得完美 (:з」∠)

  1. drawImage 覆盖图片 alpha 值为零的部分不会覆盖到背景图上，这导致输出 jpg 的时候图片颜色不对
  2. Qt 处理 RGBA 图片的时候会有图片数据的丢失的情况

  最后临门一脚的时候，发现 Qt 截取 Maya Viewport 无法批量截取多张图，
  导致合成的时候全都是最后一张图，这个问题还有待解决。

PIL 库

  原本这个库是在网上找到的，兼容 Maya 编译的 PIL 库。
  然而目前无法在 Maya2019 上使用了，估计是 Maya 版本太新了，使用了不同 VS 版本的编译工具了
